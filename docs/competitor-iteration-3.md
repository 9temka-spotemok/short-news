## Итерация 3 — Управление частотой обходов и уведомления

### Цель
- Настроить адаптивный сбор данных и много канальную систему уведомлений, учитывающую приоритет сигналов и интересы пользователей.

### Основные задачи
- **Расписание обходов**: описать конфигурацию частоты по типам источников (новости — регулярный полный обход, статичные страницы — change detection при изменении); добавить управление через Celery/воркеры, учесть ретраи, лимиты, паузы.
- **Профили источников**: поддержать режимы «Всегда обновлять» и «Только при изменении» на уровне конкурента/источника, хранить историю запусков и ошибок.
- **Система уведомлений**: очередь событий, обработчик доставки (email, Telegram, webhook), поддержка приоритетов, дедупликация, защита от двойной доставки, ведение логов.
- **Управление подписками**: API/модель для настройки частоты, каналов и типов сигналов (новости, тарифы, фичи); мультиканальные настройки (Slack/Zapier — через webhooks).
- **Frontend**: панель управления уведомлениями на `CompetitorAnalysisPage` и/или отдельной странице; отображение статуса последнего обхода, количества необработанных событий, опции настройки.
- **Документация**: обновить `README` и профильные документы (описание конфигурации расписаний, каналов уведомлений, ответственности файлов/сервисов).
- **Мониторинг уведомлений**: метрики успешной доставки, задержки, количество ошибок, интерактивные логи для аналитиков.
- **Тестирование**: интеграционные тесты расписаний и очередей уведомлений, e2e сценарий «изменение → событие → уведомление → отображение статуса», проверка защиты от дубликатов.

### Критерии готовности
- Расписания обходов управляются через API/UI и применяются без ручной правки кода.
- Уведомления доставляются по заданным каналам, логируются и не дублируются.
- Пользователи могут настраивать частоту и каналы уведомлений, видеть статус обходов.
- Мониторинг отражает ключевые метрики (успехи, ошибки, задержки).
- Документация описывает конфигурацию расписаний и интеграцию уведомлений.
